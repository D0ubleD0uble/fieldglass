# Changelog

All notable changes to Fieldglass are documented here. The format roughly follows [Keep a Changelog](https://keepachangelog.com/en/1.1.0/), and the project adheres to [Semantic Versioning](https://semver.org/spec/v2.0.0.html).

Versioning is plain [Semantic Versioning](https://semver.org/spec/v2.0.0.html), and every release is a stable one. Pre-1.0, a minor bump (`0.3.0` → `0.4.0`) may break the Rust API; a patch (`0.3.0` → `0.3.1`) does not. The same version numbers the extension and the library crates.

## [Unreleased]

### Added

- **GRIB2 spectral (spherical-harmonic) fields now decode to their coefficients.** ECMWF IFS analyses and other spectral-model output store a field as spherical-harmonic coefficients (§3 template 3.50) rather than grid values. The simpler spectral packing — `spectral_simple` (§5 template 5.50) — now decodes through `Grib2Reader::decode_spectral_message` in the Rust API, checked coefficient for coefficient against eccodes on a T63 field, mirroring the existing GRIB1 spectral support. The coefficients are not values on a grid, so such a message still does not render as a 2-D image — recovering a grid needs an inverse spherical-harmonic transform, which is a follow-up — and `decode_message_values` now reports a precise reason pointing at the spectral entry point instead of a bare "no dimensions" error. The complex form (`spectral_complex`, 5.51) is not decoded yet. Part of #302.
- **A decoded GRIB field can now be exported as CSV.** Each renderable message in the table gets an "Export CSV…" button that writes its decoded field to a file, in one of two layouts you pick: a long `lat,lon,value` table (one row per grid point) or a 2-D matrix of values. Missing points come out as empty value cells, and a large export is confirmed before it is written. Under the hood, `fieldglass-core` gains a `csv` module (`field_to_csv_matrix` / `field_to_csv_long`, formatting values in the shortest round-trippable form) and the napi bindings an `exportCsv` on the GRIB1 and GRIB2 handles. The long format needs per-point geolocation, so it is offered for the lat/lon grid family (regular lat/lon, Mercator, rotated lat/lon, Gaussian), like contours; the matrix format works for any grid with declared dimensions. Reduced-grid and projected long-format export, and NetCDF, are still to come. Part of #244.
- **GRIB2 pre-standard local image packings now decode.** Two local-use data-representation templates carry historical NCEP data with a layout identical to a registered image packing: 5.40000 (early JPEG 2000, identical to template 5.40) and 5.40010 (early PNG, identical to template 5.41). Fieldglass now decodes both through the existing JPEG 2000 and PNG codecs. eccodes ships no definition for 5.40010 and cannot decode it at all, so this is one more packing Fieldglass reads that stock eccodes cannot. Part of #307 (the ECMWF second-order local templates 5.50001 / 5.50002 remain).
- **GRIB2 log-preprocessing packed fields now decode.** Template 5.61 (simple packing with logarithmic pre-processing) stores the natural logarithm of the field under ordinary simple packing. Fieldglass now decodes it — simple unpacking followed by `Y = exp(X) − B`, where `B` is the pre-processing parameter — so these fields render instead of reporting an unsupported template. The template is experimental with no known operational producer; this completes another entry in the GRIB2 §5 packing census. Both the `B = 0` (positive field) and `B ≠ 0` branches are checked against eccodes. Closes #305.
- **GRIB2 run-length packed fields now decode and render.** JMA ships its 1 km radar, rain-gauge analysis, and nowcast products as run-length packing (DRS template 5.200): runs of quantised level indices that resolve through a level-to-value table, with level 0 marking missing. Fieldglass now decodes it, so these fields render instead of reporting an unsupported template. Because the output is one value per grid point on a regular lat/lon grid, projection, overlays, and the rest of the pipeline work with no further changes. Cross-checked against eccodes. Closes #301.
- **Click a rendered field to read its value.** A point-probe readout (#172): clicking anywhere on the image reports the geographic point under the cursor and the field's decoded value and units there — ncview-style interrogation. It reproduces the render's own per-pixel map (output pixel → lat/lon → source grid cell), so the number is exactly the one the colour shows, on every projection (source, the lat/lon-box targets, and the azimuthal/world targets — a click off the globe reports nothing). Works for GRIB and NetCDF. The optional 1-D line plot through the point is still to come. Part of #172.
- **Fields can now be compared as a difference map.** The render panel's Compare row picks a second field and an operation — `A − B`, `B − A`, `A + B`, `mean(A, B)`, or `A / B` — and renders the combined field. The difference map (`A − B`) is Panoply's flagship workflow: it reads an anomaly, a bias, or the change between two time steps at a glance. For GRIB, Field B is another message in the file; for NetCDF, it is the same variable at a second slice (its own set of dimension steppers — the two-time-steps or two-levels case). Missing data in either field is missing in the result, and a divide-by-zero drops out rather than painting a spurious value; the two fields must sit on the same grid. The combined field rides the normal pipeline, so projection, overlays, palette, scaling, and manual bounds all work on it. Off by default. Closes #239.
- **Contour lines can be drawn over any rendered field.** A Contours toggle in the render panel's Overlay row traces isolines of the field — the standard way to read a pressure or height pattern — on the same canvas as the coastlines, following the map through every projection. Levels are chosen automatically (round 1/2/5 × 10ⁿ values, à la Panoply), or set by a manual interval; the isolines break cleanly around missing data. A "contours only" mode hides the colour raster so the lines read on a blank background, and the line colour and weight are adjustable. Under the hood, `fieldglass-core` gains a `contour` module (marching squares in grid space) and the napi bindings a `projectContours` that geolocates the grid-space isolines through each grid's forward map and runs them through the same overlay-projection path the coastlines use. Contours are available for the lat/lon family (regular lat/lon, Mercator, rotated lat/lon, Gaussian); projected and reduced grids report a clear "not yet supported" message for now. Closes #238.
- **The reader API can now combine two fields into a difference, sum, ratio, or average.** The Rust `fieldglass-core` crate gains `combine_fields` and a `CombineOp` for element-wise `A − B`, `B − A`, `A + B`, `mean(A, B)`, and `A / B`, with missing propagating from either input and a divide-by-zero falling out as missing. The napi bindings expose it as `renderGridCombined` (GRIB) and `renderSliceCombined` (NetCDF), which decode two fields on the same grid, combine them, and render the result through the ordinary pipeline — so projection, overlays, palette, scaling, and manual bounds all apply to the computed field with no special casing. Fields must sit on identical grids; a mismatch is a clear error. This is the engine behind the difference-map workflow the render panel's Compare row rides on. Part of #239.
- **Fields can now be coloured on a log10 scale.** The render panel painted every field on a linear scale, which flattens quantities that span orders of magnitude — precipitation, aerosol optical depth, chlorophyll — into the bottom of the colour ramp. A new Log₁₀ toggle in the Colors row maps colour by the base-10 logarithm of the value instead, so each decade gets its own share of the ramp. Non-positive values have no logarithm and render as missing (transparent), and the colorbar keeps its labels in the true, unlogged data units. Log needs a positive floor, so the toggle is disabled with a hint until the range starts above zero, either because the field is all-positive or because a positive manual minimum is set. Linear stays the default. Closes #235.

## [0.3.0] — 2026-07-18

### Added

- **GRIB1 spectral messages now decode to their spherical-harmonic coefficients.** IFS analyses and other spectral-model output store a field as spherical-harmonic coefficients rather than grid values, and both GRIB1 spectral packings (`spectral_simple` and `spectral_complex`, the latter with its Laplacian-scaled sub-truncation) now decode through `Grib1Reader::decode_spectral_message` in the Rust API, checked coefficient-for-coefficient against eccodes on a T63 field. The coefficients are not values on a grid, so such a message still does not render as a 2-D image — recovering a grid needs an inverse Legendre transform, which is a follow-up. Until then the message reports a precise reason instead of a bare unsupported-packing error. Part of #43.
- **The reader crates are now published to crates.io.** `fieldglass-core`, `fieldglass-grib1`, `fieldglass-grib2`, and `fieldglass-netcdf` publish on each stable release, so the GRIB and NetCDF readers can be used from a Rust project with a plain version requirement instead of a git dependency. Pre-releases stay git-only, since crates.io has no pre-release channel. Closes #229.
- **Country borders, lakes, and rivers can now be overlaid on a field.** The render panel offered coastlines and a lat/lon graticule; it now adds three more Natural Earth layers. Each of the five is toggled on its own and carries its own colour and line weight, so a border reads differently from a coastline instead of both being the same thin line. Changing a layer's colour or weight repaints immediately without reprojecting. The data is Natural Earth 1:110m (public domain) and ships with the extension, so nothing is fetched at run time. Closes #237.
- **A colormap picker with eight colormaps.** The render panel used to paint every field in viridis, with no way to change it. It now offers five sequential maps (viridis, plasma, cividis, turbo, grayscale) and three diverging ones (red–blue, brown–teal, cool–warm), each of which can be reversed. Diverging maps suit anomaly and difference fields, where what matters is which side of the midpoint a value falls on. Viridis stays the default, so an existing view renders exactly as before. The colours are applied in Rust and the legend strip is drawn from the same table that paints the grid, so the two can't disagree. Closes #234.
- **Three new world map projections for global fields.** The render panel's projection picker adds Mollweide, Robinson, and Equal Earth alongside the existing equirectangular, Web Mercator, orthographic, and polar-stereographic targets. Mollweide and Equal Earth are equal-area, so a field's features cover map areas in true proportion; Robinson is the compromise projection familiar from wall maps and atlases. Each takes a central-meridian control to recentre the map. Closes #242.
- **NetCDF-4 files that store their variables in groups now show and render those variables.** Metadata resolution previously read the root group only, so a file that puts everything under groups — Sentinel-5P and other ESA Level-2 products (`/PRODUCT/...`), NASA GPM IMERG (`/Grid/...`), OMI-style HDF-EOS layouts — showed an empty or near-empty variables table even though the underlying structure was fully supported. Fieldglass now descends into every group, lists each variable and dimension with a path-qualified name (`/PRODUCT/qa_value`), resolves a variable's dimensions even when they live in an ancestor group (the netCDF scoping rule), and decodes and renders grouped variables like root ones. Checked against the Unidata `netCDF4` library. Closes #219.
- **NetCDF-4 / HDF5 files written in the "latest format" now decode more chunked variables.** Value decode previously read only the version-1 B-tree chunk index, so a file written by a recent libhdf5 (the "latest format") often reported an unsupported-layout error. Fieldglass now also reads the version-4 single-chunk and fixed-array indexes — the common cases for a one-chunk variable and a fixed-shape multi-chunk variable — the extensible-array index used for a variable with an unlimited dimension (the growable time-series case), each for both filtered and unfiltered chunks, the implicit index used for a fixed-shape, early-allocated, unfiltered variable whose chunks are stored contiguously with no on-disk index, and the v2-B-tree index used for a variable with more than one unlimited dimension. The newer version-5 layout message that libhdf5 2.0 writes for a filtered variable is read too. Everything is checked against `h5py`. Closes #216.
- **GRIB2 complex-packed fields with inline missing values now decode.** Sparse NCEP products such as the National Blend of Models (NBM) and precipitation type mark missing points inside the packed data (missing-value management, templates 5.2 / 5.3) instead of using a bitmap. Fieldglass now decodes both management modes (primary, and primary + secondary substitutes) and masks the missing points exactly like bitmap points, so these fields render instead of reporting an unsupported-template error. Fields whose groups are split row by row, which templates 5.2 / 5.3 also used to reject, decode too. Both are checked against eccodes, and the NBM file in the sample corpus now renders. Closes #217.
- **GRIB2 constant complex-packed fields now decode.** Some encoders write an all-same-value field under templates 5.2 / 5.3 as zero groups, which used to be rejected as malformed. Fieldglass now decodes zero groups as a constant field equal to the reference value, matching eccodes 2.42 and later, and the result is checked against eccodes. Closes #222.
- **Polar stereographic and Mercator WRF output now reprojects onto the map.** WRF `wrfout` rendering previously recognised only the Lambert Conformal projection (`MAP_PROJ = 1`); polar-region domains (`MAP_PROJ = 2`, polar stereographic) and tropical domains (`MAP_PROJ = 3`, Mercator) fell back to an unprojected source-only image. Fieldglass now reads both from the same global attributes and reprojects them through the projectors the GRIB paths already use, checked against independently computed coordinates. Closes #220.
- **Unrotated lat-lon WRF output now reprojects onto the map.** WRF's lat-lon projection (`MAP_PROJ = 6`), used for global and some regional domains, previously fell back to an unprojected source-only image. An unrotated domain (`POLE_LAT = 90`) is a plain regular geographic grid, so Fieldglass now geolocates it from its corner coordinates and reprojects it like any other lat/lon grid, checked against independently computed coordinates. Rotated domains (`POLE_LAT` other than 90) still render source-only, because the mapping from WRF's rotation parameters onto a rotated pole is not cleanly documented. Closes #226.

### Fixed

- **GRIB1 spectral files now open in the viewer instead of failing with "the editor could not be opened".** A spherical-harmonic (spectral) message has no grid, so its grid size and corner coordinates are absent. The message table read those absent fields with a check that missed the way the reader reports "not present", so building the table threw and the whole editor failed to open with an empty table. Fieldglass now handles the absent fields correctly: the table lists a spectral message with its packing and a dash for size and bounds, and Render is offered only for messages that have a grid. Closes #288.
- **The source projection no longer shows many grids upside-down until you check Flip Y.** The source view painted grid row 0 at the top of the image without regard to the grid's scan direction. Grids that scan south to north — common in regional Lambert and polar-stereographic output, such as the National Blend of Models — therefore came out vertically flipped, and only reprojecting to another target, or manually toggling Flip Y, set them right. The source view now orients the raster from the scan direction, so it is upright by default; Flip Y is now a genuine override rather than a routine fix. Grids that already scanned north to south are unchanged. Closes #286.
- **Grids that scan alternate rows in opposite directions no longer render as a mirrored, noisy field.** Some products, including the National Blend of Models (NBM), store their rows boustrophedon-style: the first row runs west to east, the next east to west, and so on (scanning-mode flag bit 4). The decoded values were correct, but every second row was left reversed, so the image came out as a scrambled left-right mirror of itself with a noisy seam between each pair of rows. Fieldglass now flips those rows back at the point of decode, so the field is laid out west to east on every row. The value range was already right, which is why the earlier check missed it; the fault was purely in where each point landed.
- **Overlay lines on the ±180° meridian no longer jump to the wrong edge of the map.** A coastline or graticule vertex sitting exactly on the ±180° meridian was placed on the opposite edge of the map from the one it belongs to. On Mollweide this drew a thick horizontal line clear across the map at about 71°N, through Wrangel Island, and another across Antarctica; elsewhere the stray vertex was quietly discarded instead, which nicked the Siberian coast at the ±180° meridian, dropped one small island outright, and clipped the ends of every parallel on a graticule finer than 30°. The equirectangular and Web Mercator views dropped the same vertices on a global window. Such a vertex now stays on its own edge in every view.
- **Coastlines no longer streak across a recentred world map.** When a world map projection was recentred (the central-meridian control), a coastline that crossed the map's new seam could be drawn as a line straight across the map instead of stopping at the edge — most visible on Mollweide above about 70° latitude, where recentring moves the seam onto Arctic or Antarctic coast. The overlay decided where to break a line from how far apart two points landed on screen, which misreads a seam crossing on a map that narrows toward the poles. It now breaks the line by the points' longitudes instead, so the break is correct on every projection whatever its shape.
- **Lambert and polar-stereographic grids are now placed on the Earth the message declares, not a fixed one.** The projection assumed a single spherical Earth radius (6 371 229 m) for every message. GRIB1 declares 6 367 470 m, GRIB2 declares its own shape per message, and WRF projects on 6 370 000 m — so a field could land kilometres from where it belongs. On the CMC polar-stereographic sample the far corner of the grid sat 3.5 km from where eccodes puts it, and the coastline overlay drifted from the data beneath it by the same amount. Fieldglass now reads the declared earth shape (GRIB1's flag, GRIB2's `shapeOfTheEarth`, WRF's own sphere) and projects on it, matching eccodes' point iterator across the whole grid. Oblate shapes use the spheroid's mean radius, within ~0.1 % of the true figure; true ellipsoidal projection is a follow-up. Closes #271.
- **The outermost row and column of a Lambert or polar-stereographic field no longer drop out of the render.** A grid point lying exactly on the grid's edge could come back from the projection arithmetic a fraction of a billionth of a cell outside it, which the bounds check read as off-grid and painted as background. The two projections now absorb that rounding, as the rotated lat/lon one already did.
- **A corrupt GRIB1 second-order message can no longer decode to wrong values.** The shared bit reader returns its result as a 32-bit integer, but nothing stopped a caller asking for more than 32 bits, which silently dropped the extra high bits. A malformed second-order-packed field could declare a per-group value width above 32 and hit that path. The reader now rejects a request for more than 32 bits, so such a file is reported as an error instead of rendering incorrect data.

## [0.2.0] — 2026-07-02

### Added

- **GRIB2 now decodes complex packing.** Data-representation template 5.2 splits the field into groups, each with its own reference value, bit width, and length — the GRIB2 counterpart to GRIB1 second-order packing. Fieldglass decodes the common case (general group splitting without inline missing values); the rarer row-by-row splitting and inline missing-value management report an unsupported-template error instead of mis-decoding. Closes #107.
- **GRIB2 now decodes complex packing with spatial differencing.** Data-representation template 5.3 stores the field as 1st- or 2nd-order spatial differences before grouping — the scheme NCEP GFS uses for many of its fields. Fieldglass reverses the differencing and decodes the same general-splitting case as template 5.2, checked against eccodes for both differencing orders. Closes #109.
- **GRIB2 now decodes IEEE floating-point packing.** Data-representation template 5.4 stores each value as a plain big-endian float; both 32-bit and 64-bit decode, and both are checked against eccodes. (128-bit is not supported, which matches eccodes.) This is the GRIB2 counterpart to the GRIB1 IEEE packing already shipped. Closes #110.
- **GRIB2 now decodes PNG packing.** Data-representation template 5.41 stores the field as a PNG image instead of a raw bit-packed stream. Fieldglass decodes the image with the pure-Rust `png` crate and then applies the usual reference and scale transform, so PNG-packed messages render like any other. It is checked against eccodes. Closes #118.
- **GRIB2 now decodes CCSDS / AEC packing.** Data-representation template 5.42 stores the field as a CCSDS adaptive-entropy-coding (libaec) stream, the packing ECMWF uses for its open data and many operational products. Fieldglass decodes it with a pure-Rust AEC decoder, so it needs no native libaec library and still builds for every platform, then applies the usual reference and scale transform so the messages render like any other. It is checked against eccodes. Closes #117.
- **GRIB2 now decodes JPEG 2000 packing.** Data-representation template 5.40 stores the field as a JPEG 2000 codestream, the packing HRRR, MRMS, and several other NCEP products use. Fieldglass decodes it with a pure-Rust JPEG 2000 decoder, so it needs no native OpenJPEG library and still builds for every platform, then applies the usual reference and scale transform so the messages render like any other. It is checked against eccodes. Closes #116.
- **ECMWF GRIB1 fields now show real parameter names.** Fieldglass recognises ECMWF local parameter tables 128 and 129 (originating centre 98), so IFS / ERA5 fields like 2-metre temperature, 10-metre wind, and mean sea level pressure are named instead of showing "Unknown". The tables are generated from eccodes, so they match it. Closes #48.
- **Classic NetCDF now decodes variable values.** Header parsing already listed dimensions, variables, and attributes; Fieldglass now reads a variable's data array (CDF-1 / CDF-2 / CDF-5), handling fixed-size and unlimited-dimension record variables with their interleaved on-disk layout, big-endian elements of every classic type, and `_FillValue` masking. Each variable is cross-checked against the canonical Unidata netCDF4 library, including a real NOAA ERSST sea-surface-temperature field. Groundwork for NetCDF rendering. Closes #108.
- **NetCDF-4 / HDF5 now decodes dataset values.** Fieldglass reads a NetCDF-4 / HDF5 dataset's data array through the Rust API: contiguous, compact, and chunked storage (version-1 B-tree chunk index), the deflate and shuffle filters, the dataset's element byte order, and `_FillValue` masking. Decoded values are checked against what `h5py` (libhdf5) reads from the same files. Chunked datasets that use the newer version-4 chunk index (fixed / extensible array) are not read yet and report an unsupported-layout error rather than mis-decoding. Closes #121.
- **NetCDF-4 / HDF5 files now show their dimensions, variables, and attributes in the viewer.** Fieldglass resolves the NetCDF-4 dimension-scale convention — mapping HDF5 dimension scales to named, shared dimensions, reading each variable's ordered dimension list, and telling coordinate variables apart from plain data variables and from dimensions that have no coordinate values. The metadata viewer now populates the same dimensions / variables / global-attributes tables for NetCDF-4 / HDF5 files as it already did for classic ones. Resolution covers the file's root group; variables in nested groups are a follow-up. The resolved names and dimension lists are checked against the canonical Unidata netCDF4 library (`ncdump`). Closes #174.
- **Classic NetCDF fields now render as a 2-D image.** For a classic file on a regular lat/lon grid, Fieldglass draws a 2-D slice of a variable in the render panel. It detects the latitude and longitude axes from CF conventions on the coordinate variables, lists the renderable variables, and gives a slice picker: choose the variable and the image (X / Y) axes, and step through the remaining dimensions (time, level) with per-dimension index controls. The slice synthesises a regular lat/lon grid from the coordinate arrays and feeds the existing render pipeline, so the projection picker (source, equirectangular, Web Mercator, orthographic, polar stereographic), the colormap, and the coastline / graticule overlays all work the same as for GRIB. NetCDF-4 / HDF5 and curvilinear (2-D coordinate) grids are tracked follow-ups. Closes #122.
- **NetCDF-4 / HDF5 fields now render as a 2-D image too.** The slice picker, axis detection, and synthesised-grid rendering that classic files gained now work for NetCDF-4 / HDF5 files on a regular lat/lon grid, using the dimensions and coordinate variables resolved from the dimension-scale convention. Both backings share one rendering path, so the projection picker, colormap, and overlays behave identically; the only remaining gap is curvilinear and projected grids. Closes #169.
- **WRF and GOES NetCDF fields now reproject onto the map.** Two common model outputs are regular grids in a projected coordinate system rather than plain lat/lon. WRF `wrfout` files carry a Lambert Conformal projection in their global attributes (`MAP_PROJ`, `TRUELAT1`/`TRUELAT2`, `STAND_LON`, `DX`/`DY`); GOES satellite imagery carries a CF `geostationary` grid mapping with scan-angle `x`/`y` coordinates. Fieldglass now reads both and reprojects them through the same Lambert and geostationary projectors the GRIB2 path uses, so they render with coastlines in the right place (off-disk pixels on a GOES image stay transparent). Both are cross-checked against an independent geolocation oracle. Irregular curvilinear grids (ocean tripolar, swath) remain a follow-up. Closes #168.
- **Packed NetCDF fields now render in real units.** Many NetCDF files store a field as compact integers with a `scale_factor` and `add_offset` to recover the physical values — the convention GOES, MERRA-2, and ERA5 use. Fieldglass now applies that conversion to the data it renders, so the image and the colorbar labels read in the variable's real units (kelvin, radiance, …) instead of raw integer codes. It also honours a field's `valid_range` / `valid_min` / `valid_max`, masking out-of-range points like the canonical netCDF library does. Both the classic and NetCDF-4 / HDF5 backings share one path, cross-checked against the netCDF4 library. Closes #184.
- **GRIB2 Mercator and Lambert grids now reproject onto the map.** The projection picker already reprojected GRIB2 polar stereographic grids; it now also handles GRIB2 Mercator (§3.10) and Lambert Conformal (§3.30) source grids, so products on those grids reproject into the flat targets the same way their GRIB1 counterparts do. Closes #119.
- **GRIB2 rotated lat/lon grids now reproject onto the map.** Environment Canada (HRDPS/RDPS) and DWD (ICON-EU), among others, publish on a rotated lat/lon grid (§3.1) whose coordinates sit on a tilted pole. Fieldglass now unrotates those grids so they reproject into the flat targets with coastlines in the right place, matching how eccodes places the points. Closes #120.
- **GRIB2 space-view (geostationary) grids now reproject onto the map.** Geostationary satellite imagery stored as GRIB2 §3.90 is a regular grid in the satellite's scan-angle space, which previously had no projector and could not reproject. Fieldglass adds a geostationary projector that maps each output point through the satellite's view of the Earth ellipsoid, so §3.90 grids reproject into the flat targets with coastlines in the right place and off-disk pixels stay transparent. The same projector will serve GOES NetCDF imagery in a later release. Closes #176.
- **Geostationary sector products now frame to their own extent.** When a cropped §3.90 sector (a GOES CONUS or mesoscale, or a Meteosat sector) reprojects into a flat lat/lon target, Fieldglass now derives the map frame from the sector's on-disk extent instead of defaulting to the whole satellite-visible hemisphere, so the sector fills the view rather than sitting in a hemisphere of empty space. Full-disk products, whose frame was already about right, are unchanged. Closes #181.
- **Fieldglass reads three more GRIB1 grid layouts.** Rotated/oblique lat/lon grids (`grid_type 10`, used by COSMO and other limited-area models) now parse their tilted pole and reproject like the GRIB2 rotated grids. Reduced — "quasi-regular" — lat/lon and Gaussian grids, which store fewer points toward the poles (the common ECMWF reduced-Gaussian layout), now decode their full field and render. And messages that carry no grid section at all now resolve the common predefined global grids (ON388 Table B grids 2, 3, and 4) by their catalogue number. Closes #47.
- **The globe-style projections can be recentred on any point.** The orthographic and polar-stereographic targets previously used a fixed set of centre and hemisphere presets. The render panel now takes free-form numbers: orthographic recentres on any centre longitude and latitude, and polar stereographic takes either pole with an arbitrary central meridian. Closes #113.

- **Real, attribute-heavy NetCDF-4 / HDF5 files now open.** Operational files such as NOAA GOES imagery carry enough metadata that their attributes spill out of HDF5's compact storage into a fractal heap with an *indirect* root block — a doubling table of storage blocks. Fieldglass previously stopped at the single-block case and refused these files; it now walks the doubling table, so their dimensions, variables, and attributes load like any other. Exercised by a real GOES-16 ABI file bundled in the test corpus. Part of #123.
- **Files with even more metadata now open.** When an HDF5 object has hundreds of attributes (or a group hundreds of links), the index over them is a multi-level B-tree — an internal-node tree rather than the single block smaller files use. Fieldglass previously walked only the single-level case and refused the rest; it now descends the internal nodes, so the dimensions, variables, and attributes of these larger files load too. This is the index-side counterpart to the doubling-table storage support above, and the gap a metadata-heavy operational file hits first. Exercised by a bundled fixture whose attribute index is two levels deep, cross-checked against libhdf5. Part of #33 / #123.
- **The most metadata-heavy NetCDF-4 / HDF5 files now open.** Once a file's attributes (or a group's links) overflow even the first tier of a fractal heap's storage blocks, the heap nests a second tier — *child indirect blocks* in its doubling table. Fieldglass walked only the first tier and refused the rest; it now recurses through the nested blocks, so the dimensions, variables, and attributes of these files load too. This is the storage-side counterpart to the multi-level index support above, and the next structure the metadata-heaviest operational files (MERRA-2, ERA5, CMIP6) reach. Exercised by a bundled fixture whose attribute heap nests one level deep, cross-checked against libhdf5. Part of #33 / #123.
- **Compressed (I/O-filtered) HDF5 metadata heaps now decode.** The fractal heap that holds an object's dense attributes (or a group's dense links) may have its storage blocks run through a compression filter. Fieldglass previously stopped at unfiltered heaps and refused a filtered one; it now reverses the heap's filter pipeline — deflate and shuffle, the same filters it already applies to dataset chunks — before reading the blocks, so a conforming file that uses one loads its dimensions, variables, and attributes like any other. Both the single-block and doubling-table heap layouts are covered. Part of #33.

### Fixed

- **GRIB1 second-order fields that are entirely constant now decode.** When every group in a second-order message has zero width, the file holds no second-order values and the pointer to them lands just past the end of the section. Fieldglass treated that as malformed and refused the message; it now decodes, matching eccodes. Closes #91.
- **Malformed GRIB1 files can no longer crash the reader.** A message that declared a length shorter than its own header, or that ended partway through a section's length field, used to panic instead of reporting a parse error. Both cases now surface a clean error. The crashes were found by a new fuzz target that exercises the GRIB1 decode path against arbitrary bytes, run as a time-boxed CI job. Closes #92.
- **Malformed GRIB2 files can no longer crash the reader.** Several crafted inputs used to crash instead of reporting a parse error: a message whose declared total length overflowed a 64-bit offset, one whose bit-map or data section declared a length running past the end of the message, and one whose grid dimensions disagreed with the grid's own declared point count (which could make decoding try to allocate gigabytes for a file that held no such data). All of these now surface a clean error. The crashes were found by a new fuzz target that exercises the GRIB2 decode path against arbitrary bytes, run as a time-boxed CI job. Closes #132.
- **Truncated GRIB1 matrix-of-values fields now report an error instead of mis-decoding.** When a matrix message's packed data ran short of what its secondary bitmap declared, the decoder filled the gap with missing values and shifted every later value by one. It now surfaces a clean parse error, matching how the other packings handle a short data section.
- **Hardened parsing of untrusted files.** A NetCDF header name length is now bounded by a fixed cap rather than the file size, a GRIB2 end-of-message check guards its own bounds locally, and the projection code sorts with a total order so a stray non-finite value can't panic a render. None of these change how valid files decode.
- **Small NetCDF attribute values now show in full.** The metadata viewer rounded numeric attributes to six decimal places, so a tiny value like a GOES `scale_factor` of about 0.00000067 displayed as `0.000001`. Numeric attributes now show their exact stored value. Closes #184.
- **NetCDF fields that mark gaps with `missing_value` now mask them.** Decode masked only points equal to a variable's `_FillValue`; it now also masks the CF `missing_value` attribute, the older convention many products still use, so those gaps read as missing instead of as a real data spike. Both the classic and NetCDF-4 / HDF5 backings handle it, cross-checked against the netCDF4 library.
- **Global grids that start at the antimeridian now reproject correctly.** A regular lat/lon grid whose longitudes run eastward from 180° — the layout ECMWF open data uses — rendered mirrored in the equirectangular view and collapsed to a thin sliver at the seam in the orthographic and polar-stereographic views. Its longitude span was measured as a single grid step instead of the full globe. The reprojection now unwraps a grid that crosses the antimeridian, so it renders like a 0°-origin grid. The same fix covers Mercator and Gaussian source grids. Two related hardenings ride along: the rare grid that scans east-to-west now stays in its own coordinates instead of reprojecting wrongly, and a file with a corrupt (non-finite) grid corner now reports a clear error instead of rendering misplaced data.
- **Global grids no longer show a hairline gap at their wrap meridian.** A global grid stores each longitude once, so the last fraction of a degree between its final column and the first belongs to neither — the orthographic and polar-stereographic views drew it as a one-pixel seam line. The renderer now recognises a grid that spans the full globe and samples across the wrap, interpolating between the last and first columns, so the seam is continuous in every projection.
- **ECCC rotated grids no longer render split across the map.** The HRDPS continental grid's columns cross the antimeridian of its rotated frame, and the map window computed for it swept the wrong way around the globe — a nearly global window with the field split between its left and right edges. The window now follows the grid's true span, so the domain frames correctly in the equirectangular and Web Mercator views.
- **The render tab keeps its settings when you switch tabs.** Switching to another editor tab and back rebuilt the render panel with default selections — projection, centre, resampling, color range, bounds, overlays, and the NetCDF variable and slice pickers all reset. The panel now remembers its selections and restores them when its tab is shown again, repainting the image the way you left it.

- **Lambert grids with a 0–360 reference longitude no longer render blank.** A longitude wrap-around error projected every point off the grid when a file stored its `LoV` in the 0–360 range, as NCEP and Eta files do. These grids now reproject correctly.

- **Coastlines and the grid render cleanly at view edges.** A longitude wrap error used to split overlay lines at the edge of a regional or manually-windowed view, leaving a notch. Long lines whose endpoints both fall off-screen but whose path crosses it are now drawn, and the dateline split now applies only to views that actually wrap.

- Versioning: dropped the `-beta.1` semver pre-release suffix in favour of the VS Code pre-release convention. The Marketplace rejects semver pre-release tags; future betas will be `0.1.1`, `0.1.2`, … with stable jumping to `0.2.0`.

### Notable display fixes

- GRIB1 GDS lat/lon decode now correctly handles WMO sign-and-magnitude encoding (was previously two's-complement, producing bogus bounds like `-8298.608°` for a valid `-90°` south pole).
- Level rendering is unit-aware: pressure levels show as `200 hPa`, height-above-ground as `2 m`, layer types as ranges (`100 – 85 kPa`), with `—` for fixed-surface types where the value byte has no meaning.
- Forecast time display respects the WMO time-range indicator: `+24h`, `0–24h accum`, `analysis`, etc., instead of collapsing to a raw P1 byte.
